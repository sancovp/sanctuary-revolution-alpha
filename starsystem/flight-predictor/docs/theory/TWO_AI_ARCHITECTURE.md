# Two-AI Architecture

## The Separation

There are two distinct AIs in this system, both embedding-based but with different roles:

### LLM (Meta-Sequence Generator)

- Takes observable context
- Generates token predictions about it
- Not just sequence generation - **context-aware** sequence generation
- A higher-order meta-sequence generator
- Provides the **semantic lift** (natural language → structured data)
- Performs **collapse** (superposition → specific instantiation)

### Planning AI (Capability Predictor)

- Takes structured observations (generated by LLM via tool schema)
- Outputs well-known IDs of capabilities (tool/skill names)
- Uses semantic similarity search over **typed hypergraph**
- Each capability has **neighborhood embeddings** (cloud of senses)
- Provides **superposition** (potential solution space)

## The Data Flow

```
┌─────────────────────────────────────────────────────────────┐
│ LLM TRACK (semantic generation required)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Parse user intent → structured PlanStep[]                │
│ 2. Fill tool arguments (requires understanding context)     │
│ 3. Decide next action when ambiguous                        │
│ 4. Generate natural language outputs                        │
│ 5. COLLAPSE superposition via emergent morphisms            │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ (structured data handoff)
┌─────────────────────────────────────────────────────────────┐
│ SYSTEM TRACK (mechanical, no LLM needed)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. predict(structured_intent) → hypergraph of capabilities  │
│ 2. record(what_LLM_actually_used) → usage log               │
│ 3. compare(predicted vs actual) → mismatch signal           │
│ 4. train(usage_sequences) → learn chains                    │
│ 5. compile(frequent_chains) → named programs                │
│ 6. surface(compiled_programs) → back to LLM as options      │
└─────────────────────────────────────────────────────────────┘
```

## Why Two AIs?

The LLM does **semantic lift** - it understands meaning, context, intent. This requires massive training on human knowledge.

The Planning AI does **structural matching** - it finds capabilities based on semantic neighborhoods. This is mechanical once the hypergraph is built.

Keeping them separate means:
- Planning AI can be fast (just embedding lookups + graph traversal)
- Planning AI can be trained on YOUR usage patterns (not general knowledge)
- LLM focuses on what it's good at (understanding, generation)
- The loop between them creates compound intelligence

## The Dance

1. User expresses intent
2. LLM parses → structured query
3. Planning AI returns **hypergraph** (superpositioned capabilities)
4. LLM **collapses** by generating morphisms for each match
5. Execution happens
6. Observation feeds back to Planning AI
7. Planning AI improves predictions
8. Next cycle is better
