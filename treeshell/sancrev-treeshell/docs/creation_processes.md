# Creation Processes - How Things Get Made

## The Four Creations

| Thing | What It Is | When Created |
|-------|-----------|--------------|
| **PAIA** | Personal AI Agent - the Vehicle | First, foundation |
| **MVS** | Minimum Viable Sanctuary - first playable | After basic PAIA exists |
| **SJ** | Sanctuary Journey - the hero's path | Emerges through play |
| **VEC** | Victory Emanation Crystal - proof of transformation | End of journey |

---

## 1. PAIA Creation Process

**What:** The Vehicle - your compound intelligence system

### PAIA vs Substrate (Critical Distinction)

```
SUBSTRATE (e.g., Claude Code)
├── Raw code agent capability
├── Can read/write files, run bash, search
├── Responds to prompts
├── NO autonomy - waits for user
├── NO memory across sessions
├── NO self-improvement
└── = A tool

PAIA (the exoskeleton ON the substrate)
├── At least ONE loop (autopoiesis, guru, etc.)
│   └── Makes it run autonomously
├── Adapts to substrate (Claude Code today, others tomorrow)
├── Has memory (STARLOG, CartON, skills)
├── Has self-improvement (crystallization, goldenization)
├── Has identity (persona, frame)
└── = YOUR agent
```

**The formula:**
```
PAIA = Substrate + Loop + Memory + Identity + Knowledge Graph

Claude Code alone = substrate (powerful tool, not YOUR agent)
Claude Code + autopoiesis + STARLOG + persona + CartON = PAIA
```

**Minimum PAIA:**
```
□ ONE loop (autopoiesis minimum - makes it self-continuing)
□ ONE memory system (STARLOG minimum - survives sessions)
□ ONE identity (persona minimum - knows who it is)
□ ONE CartON identity (knowledge graph - knows what it knows)
= Can run autonomously, remembers, has purpose, has knowledge
```

Without a loop, it's just Claude Code with extra prompts.
**The loop is what makes it a PAIA.**

### CartON Identity Structure

The PAIA's CartON identity is its self-knowledge:

```
PAIA CartON Identity
├── STARLOG Projects Collection
│   ├── project_1/ (starlog data)
│   ├── project_2/ (starlog data)
│   └── ... (all projects the PAIA has worked on)
├── Personas Collection
│   ├── persona_1 (with its own CartON identity)
│   ├── persona_2 (with its own CartON identity)
│   └── ...
├── Observations
│   ├── What the PAIA has learned
│   ├── Patterns discovered
│   └── Insights crystallized
└── Relationships
    ├── Project → Persona mappings
    ├── Skill → Domain mappings
    └── Cross-project patterns
```

**Collection = Domain community for X**
- STARLOG projects collection = all projects this PAIA knows about
- Personas collection = all personas this PAIA can be
- Skills collection = all skills this PAIA has

**HOME mode reads from CartON identity:**
- LEARN: Query the knowledge graph
- CONFIGURE: Update the graph (add rules, skills)
- DECIDE: Choose from known projects/personas

### Personas as Gear (Optional Category)

Personas are OPTIONAL gear you can equip on a PAIA:

```
PAIA Persona System
├── Baseline: PAIA has ONE core identity (always)
├── Optional: Can equip personas (mode shifts)
├── Personas ≠ Subagents
│   ├── Subagent = spawn separate agent (delegation)
│   └── Persona = change who PAIA is being (transformation)
└── Can be turned on/off
```

**Persona vs Subagent:**

| Aspect | Persona | Subagent |
|--------|---------|----------|
| What | Mode of being | Separate agent |
| Where | Same context | Spawned process |
| Memory | Shares PAIA memory | Own context |
| Identity | "I am now X" | "X is helping me" |
| Use case | Domain expertise shift | Parallel/isolated work |

**Persona as Gear:**
```
PAIA Gear Slots
├── Skills (knowledge)
├── MCPs (tools)
├── Hooks (triggers)
├── Flights (procedures)
├── Loops (autonomy patterns)
└── Personas (identity modes) ← Optional gear category
    ├── Can have 0 personas (just baseline PAIA)
    ├── Can have N personas (equip as needed)
    └── Each persona = frame + skillset + MCP set + CartON identity
```

**When to use personas:**
- Different domains need different expertise → equip domain persona
- Same PAIA, different "hats" → persona per hat
- Want to delegate? → subagent
- Want to transform? → persona

## 2. MVS Creation Process

**What:** Minimum Viable Sanctuary - first playable game state

**MVS = PAIA + Game Loop + Journaling**

The MVS is when your PAIA becomes a "game" you play daily:
- Morning journal → Day mode
- Work sessions → JourneyLogs
- Evening journal → Night mode
- Night flights → Autonomous work

**MVS Requirements:**
```
MVS Checklist
=============

□ PAIA exists (even L1 with minimal subsystems)
□ Journaling works (sanctuary_system MCP)
□ Day/Night cycle understood (even if manual)
□ At least ONE work loop defined
□ HOME mode accessible (can learn, configure, decide)

= You can "play" the game daily
```

**MVS Creation Flow:**
```
1. Have a PAIA (any level)
2. Set up sanctuary_system MCP for journaling
3. Do ONE morning journal entry
4. Do ONE work session (plot_course → SESSION → LANDING)
5. Do ONE evening journal entry
6. Reflect: "Did I feel sanctuary or wasteland?"

If sanctuary → MVS achieved, keep playing
If wasteland → identify what's missing, fix, retry
```

## 3. SJ (Sanctuary Journey) Creation Process

**What:** Your hero's journey through sanctuary - emerges through play

**SJ is NOT built, it's DISCOVERED.**

The journey emerges from:
- JourneyLogs (obstacle → pathway → overcome)
- EpisodeArcs (sessions become episodes)
- JourneyArcs (episodes become story arcs)
- EpicArcs (arcs become epic transformation)

**SJ Emergence Flow:**
```
Play MVS daily
    ↓
JourneyLogs accumulate
    ↓
NarrativeManager creates EpisodeArcs
    ↓
Patterns emerge (recurring obstacles, breakthroughs)
    ↓
JourneyArc crystallizes (the story of this mission)
    ↓
Multiple JourneyArcs → EpicArc (the hero's transformation)
    ↓
SJ = the documented path from Call to Return
```

**SJ Documentation:**
```
SJ Structure
============

1. THE CALL
   - What pulled you into this journey?
   - What was the ordinary world before?

2. THRESHOLD CROSSING
   - When did you commit?
   - What did you leave behind?

3. TRIALS (from JourneyLogs)
   - Key obstacles faced
   - Pathways discovered
   - How you overcame

4. TRANSFORMATION
   - What changed in you?
   - What capabilities emerged?
   - What do you now know/do that you couldn't before?

5. RETURN
   - What do you bring back?
   - How does it serve others?
   - = VEC
```

## 4. VEC (Victory Emanation Crystal) Creation Process

**What:** Proof of transformation - crystallized output that serves others

**VEC = Deliverable that proves the journey happened**

Not just "I learned something" but "I built something that works and others can use."

**VEC Types:**
```
VEC Categories
==============

Code VECs:
- Published package (pip install X)
- Working MCP server
- Golden flight config
- Deployed automation

Content VECs:
- Blog post teaching what you learned
- Video walkthrough
- Documentation others can follow

Product VECs:
- Sellable plugin
- SaaS feature
- Consulting offering

Identity VECs:
- New persona (with proven capabilities)
- New skillset (with golden skills)
- Reputation artifact (proof of expertise)
```

**VEC Creation Flow:**
```
1. Complete SJ (or significant portion)
2. Ask: "What did I create that WORKS?"
3. Package it for others:
   - If code → publish, document, test
   - If content → polish, publish, promote
   - If product → price, sell, support
4. Validate externally:
   - Did someone else use it?
   - Did it work for them?
   - = VEC proven
```

**VEC Quality Test:**
```
A true VEC survives these questions:

1. Can I point to it? (URL, package, artifact)
2. Does it work without me? (Autonomous value)
3. Has someone else benefited? (External validation)
4. Does it prove my transformation? (Not just completion)

If all YES → VEC is real
If any NO → Still crystallizing
```

## The Full Flow

```
PAIA (build the Vehicle)
    ↓
MVS (Vehicle becomes playable game)
    ↓
SJ (play the game, journey emerges)
    ↓
VEC (crystallize proof of journey)
    ↓
PAIA leveled up (VEC = legendary component)
    ↓
Repeat at higher level
```

## Key Insight

**PAIA is the substrate. MVS is the game. SJ is the story. VEC is the proof.**

You can't have MVS without PAIA.
You can't have SJ without playing MVS.
You can't have VEC without completing (part of) SJ.

Each VEC levels up the PAIA, making the next MVS more powerful, enabling harder SJ, producing bigger VEC.

**This is the compounding loop.**
