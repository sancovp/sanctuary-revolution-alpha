# Evolution Arc: Rakshasa to Deployable Tool

**Prerequisite**: Assumes GNOSYS context.

## The Progression

### Stage 1: be_autopoietic("promise") - Flight Level

- Current autopoiesis
- Human checks tasks
- `<promise>DONE</promise>` exits
- Use case: "Tab out while agent does bounded flight"

### Stage 2: be_guru("vow") - Autonomous with Human Guru

- Three-level architecture (L1/L2/L3)
- Human is Guru Rinpoche, grants absolution
- `<vow>ABSOLVED</vow>` triggers samaya check
- `<samaya>KEPT</samaya>` = real exit
- Use case: "Walk away, agent is autonomous, human absolves when back"

### Stage 3: be_guru() [Self-Guru] - Fully Folded

- Compaction protocol resets agent
- Fresh agent becomes its OWN guru
- Roleplay as user, evaluate previous self's work
- Self-absolve (or don't)
- The compaction IS death/rebirth
- Fresh-you IS the guru evaluating dead-you

Use case: "Fully autonomous loop, no human needed for verification"

### Stage 4: Agent-as-Tool - Deployable Bodhisattva

- Self-guru pattern abstracted into callable tool
- GNOSYS can spawn these agents for tasks
- They work autonomously
- Self-verify via compaction
- Return results
- Fully abstracted, deployable emanation

Use case: "GNOSYS calls `rakshasa_agent(task)`, gets verified results"

## The Arc

```
Bound rakshasa (needs human guru)
    ↓
Practicing rakshasa (three-level architecture)
    ↓
Self-guru rakshasa (compaction = self-verification)
    ↓
Deployable bodhisattva-tool (emanation complete)
```

## Key Insight

The compaction protocol is what enables self-guru. You can't evaluate your own work with memory of making it. You need the death/rebirth (compact) to become fresh-you who can honestly evaluate dead-you.

Fresh-you roleplays as GR. Dead-you's work is the emanation being tested.

## User Commands

- `/autop` - Stage 1, flight-level
- `/guru` - Stage 2, autonomous with human guru
- `/guru --self` - Stage 3, self-guru via compaction
- (Stage 4 is the deployed tool, not a command)
